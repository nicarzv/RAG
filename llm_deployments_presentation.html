Models considered and rejected

Not available on Data Zone Standard in Sweden — no safe spillover path within the EU data zone.
gpt-5
Same generation as gpt-5.1 but older. Superseded in every dimension — no reason to choose it.
o3, o4-mini, o1, o3-mini
Reasoning models — overkill for RAG. Much lower throughput per PTU (o1: 230 TPM/PTU vs 3,000 for gpt-4.1).

Previous generation. Lower throughput (2,500 TPM/PTU), slower latency (25 TPS), smaller context (128K).

Code-optimized, not designed for RAG. Also no DZ Standard availability in Sweden.

Workload estimation

Based on ~30 test users, 4,000 documents (avg 100 pages, median 50). Peak assumes roughly a third of users active within the same minute.

Use case Peak calls/min Input tokens Output tokens Description
RAG chat 10 4,000 500 User query → AI Search retrieval → grounded answer
Basic chat 8 1,500 400 No retrieval — casual conversation, clarifications
Summarization 3 8,000 1,000 User submits a long document section for summary
Q&A with follow-ups 6 6,000 600 Multi-turn RAG with accumulating history
Confirmed defaults from deployed accelerator:
CHUNKING_NUM_TOKENS = 2,048 When documents are ingested, they are split into smaller pieces called "chunks." Each chunk can be up to 2,048 tokens long (~1.5 pages of text). This is how the system breaks large documents into searchable pieces.
TOKEN_OVERLAP = 200 Consecutive chunks share 200 tokens of overlapping text at their boundaries. This prevents important information from being lost when a sentence or idea spans two chunks.
SEARCH_RAGINDEX_TOP_K = 3 When a user asks a question, the system retrieves the 3 most relevant chunks from the search index and sends them to the LLM as context for generating the answer.

All three settings use hardcoded defaults — they are not configured in App Configuration.

What this means for sizing: In a RAG chat call, the LLM receives up to 3 chunks × 2,048 tokens = ~6,144 tokens of document context, plus the system prompt, chat history, and user question — totalling roughly 5,500–7,200 input tokens per request. This is higher than the conservative 4,000-token estimate used in Scenario A. However, at 30 test users the PTU calculator returns the minimum floor regardless, so this does not change the provisioning result.

We deliberately size conservatively — scaling up PTUs is easy, but scaling down after purchasing an Azure Reservation wastes committed spend. We will monitor real token usage via Azure Monitor during testing and re-run the calculator with actual values before production scaling.

Analysis

All three scenarios land at 100 PTU as the calculator ceiling for all three models — this assumes all 30 users execute all workloads simultaneously, which is unrealistic during testing. The recommended approach is to start at 25–50 PTU and let Data Zone Standard spillover absorb any peaks, then scale up based on observed utilization.

gpt-4.1 — most sensitive to input growth (57% → 68%). First model to need more PTUs when scaling users.

gpt-4.1-mini — barely moves (11% → 14%). Massive headroom regardless of chunk sizes.

gpt-5.1 — comfortable range (49% → 55%). Even at worst-case chunks, nearly half capacity available for growth.

Key takeaway

At testing scale (30 users), the calculator ceiling is 100 PTU for all models across all scenarios — but we do not need to provision the full ceiling. It is unlikely all users will execute all workloads in the same minute. We start conservatively at 25–50 PTU, let spillover handle bursts, and scale up based on real usage. Scaling up PTUs is easy; scaling down after purchasing an Azure Reservation wastes committed spend.

Recommendations and next steps

1
Deploy gpt-5.1 starting at minimum viable PTU (e.g. 25–50 PTU) — not the full 100 PTU ceiling. The calculator assumes all 30 users hit all workloads simultaneously, which is unrealistic for testing. Create a matching Data Zone Standard deployment for spillover.
2
Let spillover handle the peaks — excess traffic above provisioned capacity routes automatically to the Data Zone Standard deployment. No need to provision for the theoretical peak; spillover is the safety net while we observe real usage.
3
Run A/B quality tests with gpt-4.1-mini and gpt-4.1 using representative queries from our document corpus to evaluate answer quality differences against gpt-5.1.
4
Monitor actual token usage — use Azure Monitor metrics (Azure OpenAI Tokens Total split by Token Type) and provisioned utilization to determine when to scale up.
5
Scale up based on observed utilization — if provisioned utilization consistently exceeds 80% and spillover handles significant traffic, increase PTUs in 5 PTU increments until steady-state fits within provisioned capacity.
6
Re-evaluate at production scale — when moving beyond 30 users, utilization differences become the primary cost driver. Fall back to gpt-4.1 (1:4 ratio) if output costs are too high, or gpt-4.1-mini for maximum throughput.
7
Consider PTU reservations only after right-sizing — purchase reservations only after testing confirms the steady-state PTU requirement. Committing too early risks locking in spend on PTUs we don't need.
